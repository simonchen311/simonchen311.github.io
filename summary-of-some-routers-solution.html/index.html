iOS开源路由技术总结

开发
  • Source Code
  • iOS

问题的引出

在团队开发中，一个产品往往会分割为许多不同的模块，交给不同的开发人员，而模块之间如果有交互需求的话，必然会需要了解其他模块所暴露的调用接口，传参方式等等，从而引入其他模块头文件，当产品越来越庞大，所依赖模块不断增多的情况下，外部模块如果发生一些变动很有可能会导致需要跟着它的变化修改大量代
码，这不符合开发中“高内聚，低耦合”的基本原则。

设计一个路由层是可以用来解决这个问题的，其目的就是解决各模块之间的交互需要引入其他模块的头文件，通过路由中转来解除模块间的相互依赖。

目前已经有了一些开源的路由解决方案，本次我选取了在Github上几个典型的开源路由： HHRouter、Routeble-iOS、JLRoutes、MGJRouter、CTMediator，阅读源码并总结学习这些路由方案是如何实现的。

大部分的路由解决方案都是通过URL Scheme调用方式进行的，通过先前约定好的URL格式来利用路由进行模块间的调用。而一个完整的URL组成部分如下所示：

Scheme://username:password@hostname[:port]/path/[;parameters][?query]#fragment

HHRouter对外暴露了4个匹配接口。

matchController用来根据传入路径匹配路由获取对应的viewController

matchBlock和callBlock都是通过匹配路由获取block，两者区别是matchBlock方法需要手动调用block，而后者在方法内部自动进行调用了。

路由匹配的核心部分是将传入的URL进行解析，其过程分为几个步骤：

  • 获取URL路径部分并同样进行分割，将路径每个组成部分分别去routes路由中找，看能否完全匹配
  • 如果有完全匹配的路由路径，则获取URL的参数部分，将参数保存在param字典中
  • 最后根据理由匹配的结果，将viewController取出或者block取出，保存在param字典中
  • 将param字典返回

Routable-iOS

Routable内部还封装了一些子类用来配合整个路由工作

UPRouterOptions

UPRouterOptions用来配置一些路由选项

其内部封装的成员属性包括：

  • openClass ：路由匹配的视图控制器类
  • callback ：路由匹配的执行回调
  • modal ： 该属性用来确定匹配的UIViewController是以模态视图形式打开还是将其压入当前的视图控制容器栈
  • presentationStyle：弹出viewController时的风格选项
  • transitionStyle：模态视图弹出的动画风格
  • defaultParams：默认参数，以字典封装
  • shouldOpenAsRootViewController：该属性用来确定匹配的UIViewController是否作为根视图

UPRouterOptions暴露的接口都是用来创建返回一个UPRouterOptions实例但其配置属性值不同。

UPRouter

UPRouter路由类中封装了两个字典

  • routes: key是路由规则，value是UPRouterOptions路由配置选项
  • cachedRoutes: key是最终的路由URL，value是RouterParams

路由类中同样封装了多个map方法来配置路由规则，在代码中就可以看出传入的NSString就是路由规则，其作为key值存放在routes数组中

路由匹配时需要解析传入的URL参数，从代码可以看出，整个解析过程；

  • 先检查cachedRoutes是否有对应的url记录，如果有则直接取出value并返回
  • 分割url，获取每个组成部分
  • 遍历routes路由规则字典，取出routes的路由路径，分割后与上一步获取的组成部分进行一一匹配
  • 匹配成功后创建RouterParams对象，并将其作为value存放在cachedRoutes字典中。

如果匹配到的是UIViewController，需要将匹配封装出来的RouterParams中转换成对象的Controller
这里有个注意的是匹配的UIViewController需要实现+allocWithRouterParams：或-initWithRouterParams：方法
