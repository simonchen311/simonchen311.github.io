FBKVOController源码阅读

开发
  • Source Code
  • iOS


定义

键值观察(KVO)在苹果的官方文档中是这么定义的：它是一种允许观察者对象在所观察其他对象的属性发生改变时可以获得这些改变的通知这样一种机制。

但是原生的KVO在实现上存在着这么一些问题：

  • KVO在添加观察者后，在不再需要继续观察的时候应该将其移除。
  • 添加和移除需要成对出现，我们在开发的时候往往可能会由于疏忽导致代码里忘记在不需要使用的时候去释放掉观察者，那么这样就会埋下程序会crash的隐患。
  • KVO的所有获取改变通知有着统一的回调函数进行处理，如果观察者同时观察了多个属性的变化，那么这个回调可能会变得非常复杂。

FBKVOController是Facebook开源的一套基于KVO再封装的解决方案，相比较原生的KVO，它有如下这些优点：

  • 在注册观察者的时候提供了多种形式处理回调：block或者指定一个处理回调方法等等，从而达到了分散回调处理方法的目的。
  • 不再需要手动移除观察者了，FBKVOController会在内部自动在观察者controller释放掉的时候进行移除处理。
  • 在多线程环境下安全。

下面针对FBKVOController这个开源解决方案来继续的阅读其源码，看看其是如何实现这些改进的。

从其提供的接口可以看出，在注册观察者时可以分别传入一个block来处理观察通知回调，也可以指定一个SEL来处理，同样也提供与原生一致的NSKeyValueObserving回调来处理。

同时FBKVOController也可以一次传入多个keyPath来一次性注册观察，从未减少了很多代码的编写量来依次单个注册。

NSDictionary提供了Key-to-Object的映射，并且Key必须是一个遵守NSCopying协议的不可变对象，如NSString，因为在添加到字典中的键值对，字典始终会对key进行一次拷贝，从而复制到其私有的位置。

NSMapTable提供的是一种一般意义上的映射，它与字典不同的是，它支持Object-to-Object的映射；在创建NSMapTable对象时会提供一些选项来决定Object的管理方式，无论是作为键还是值的Object都需要指明：
NSMapTableStrongMemory (a “memory option”)：添加后其引用计数+1
NSMapTableWeakMemory (a “memory option”)：不会修改引用计数，并且在对象释放后，会将其从容器中移除
NSMapTableObjectPointerPersonality (a “personality option”)使用移位指针(shifted pointer)来做hash检测及确定两个对象是否相等
NSMapTableCopyIn (a “copy option”)：添加的是复制的一份新的对象于集合中
