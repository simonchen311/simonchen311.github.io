Facebook开源代码 - FBKVOController阅读

定义

键值观察(KVO)在苹果的官方文档中是这么定义的：它是一种允许观察者对象在所观察其他对象的属性发生改变时可以获得这些改变的通知这样一种机制。

但是原生的KVO在实现上存在着这么一些问题：

  • KVO在添加观察者后，在不再需要继续观察的时候应该将其移除。
  • 添加和移除需要成对出现，我们在开发的时候往往可能会由于疏忽导致代码里忘记在不需要使用的时候去释放掉观察者，那么这样就会埋下程序会crash的隐患。
  • KVO的所有获取改变通知有着统一的回调函数进行处理，如果观察者同时观察了多个属性的变化，那么这个回调可能会变得非常复杂。

FBKVOController是Facebook开源的一套基于KVO再封装的解决方案，相比较原生的KVO，它有如下这些优点：

  • 在注册观察者的时候提供了多种形式处理回调：block或者指定一个处理回调方法等等，从而达到了分散回调处理方法的目的。
  • 不再需要手动移除观察者了，FBKVOController会在内部自动在观察者controller释放掉的时候进行移除处理。
  • 在多线程环境下安全。

下面针对FBKVOController这个开源解决方案来继续的阅读其源码，看看其是如何实现这些改进的。

源码分析

FBKVOController的结构很简单，其核心代码包括如下几个类：

  • FBKVOController
  • NSObject+FBKVOController

NSObject+FBKVOController

NSObject+FBKVOController.m是一个分类。通过关联对象给NSObject提供一个Retain和nonRetain的KVOController。

RunLoop探究

以前对RunLoop的认知一直是停留在非常模糊的阶段，只是知道其与线程相关，维持线程的运转不退出，趁现在任务不紧，花点时间研究下RunLoop的源码，看其作用以及是如何实现的。

概念

一个线程一般只执行完一个任务后就退出，但是RunLoop能执行这么一种机制：让线程可以随时处理事件且处于不退出的状态，处理完成后线程处于休眠状态，当有事件的时候会被唤醒。

RunLoop本身是不能够直接来创建的，除了主线程的RunLoop，其他线程默认是不开启的，只有在获取当前线程的RunLoop才会被创建，如果不主动获取，那么它一直都不会有。可以通过下面两套机制来获取当前线程的RunLoop，其中NSRunLoop实际上是对CFRunLoop的封装。

NSRunLoop

CFRunLoopRef

RunLoopMode

一个RunLoop内部包含了多个RunLoopMode，RunLoop同一时间只能且必须在一种特定的Mode下运行

  • NSDefaultRunLoopMode(kCFRunLoopDefaultMode)：默认Mode
  • UITrackingRunLoopMode：用于UI跟踪的Mode，在ScrollView滑动时，为了保证UI顺畅度而切换到当前的Mode中来
  • UIInitializetionRunLoopMode：在刚启动App时进入的第一个Mode，之后不会再使用
  • GSEventReceiveRunLoopMode：接受系统时间的内部Mode
  • NSRunLoopCommonModes(kCFRunLoopCommonModes)：占位Mode，不是一种真正意义的Mode。

给App增加iTunes文件共享功能

可是手头的项目要测试，需要提取出应用中的文档，于是只能使用iTunes的文件共享功能来提取了，这需要进行一步简单的设置：

在Xcode中，在plist配置文件中增加“Application supports iTunes file sharing”，并将这一项的值设置为YES即可。

之后在真机上运行App，并在iTunes上查看，就可以共享文件给App或者导出App中的文件了。

AOP编程 - Aspects源码阅读

概念

AOP: Aspect Oriented Programming 面向切面编程。利用AOP可以对业务逻辑的各个部分进行隔离，提取模块可能都要重复的部分，从而使得业务逻辑各部分之间的耦合度降低，提高程序的可重用性，同时提高了开发的效率。iOS中利用运行时，动态地将代码切入到类的指定方法、指定位置上的编程思想就是面向切面的编程。

利用AOP，可以将日志记录，性能统计，安全控制，事务处理，异常处理等代码从业务逻辑代码中划分出来，通过对这些行为的分离，我们希望可以将它们独立到非指导业务逻辑的方法中，进而改变这些行为的时候不影响业务逻辑的代码。

Aspects是AOP思想在iOS下的实现，写这篇文章主要是阅读Aspects源码，查看它具体实现过程。

KVC与KVO


概述

KVC是一种通过字符串名称间接访问对象属性的方法。其方法在NSKeyValueCoding中被声明，默认的实现方法由NSObject提供。非对象参数和返回类型会被识别自动封装、解封。

通过KVC这种间接访问方法，开发者就能在运行时确定属性的绑定，而不是在编译时。


键和键路径

  • key： 键是一个标识了对象某个属性的字符串。通常在接收的对象中，一个键对应的是一个访问方法或者实例变量的名称。
  • keyPath： 键路径由一个或多个键组成，以点做分隔符，用于指定一个对象属性遍历序列。

注意：objectivec中的点语法是调用属性的存取方法，而键路径中的点是用以界定元素。

iOS Runtime 运行时笔记(四)

对于Runtime，还有一个经常使用的到的，那就是关联对象了

定义

关联对象是指两个对象关联起来，其中一个对象作为另一个对象的一部分；它允许开发者为任何对象附着键值数据。

可以给某对象关联许多其他对象，这些对象通过“键”来区分。存储对象值的时候，可以指明“存储策略”，用以维护相应的“内存管理语义”。设置关联对象用的键是个“不透明的指针”，即所指向的数据结构不局限于某种特定类型的指针。
