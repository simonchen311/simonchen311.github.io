AOP编程 - Aspects源码阅读

开发
  • Source Code
  • iOS

概念

AOP: Aspect Oriented Programming 面向切面编程。利用AOP可以对业务逻辑的各个部分进行隔离，提取模块可能都要重复的部分，从而使得业务逻辑各部分之间的耦合度降低，提高程序的可重用性，同时提高了开发的效率。iOS中利用运行时，动态地将代码切入到类的指定方法、指定位置上的编程思想就是面向切面的编程。

利用AOP，可以将日志记录，性能统计，安全控制，事务处理，异常处理等代码从业务逻辑代码中划分出来，通过对这些行为的分离，我们希望可以将它们独立到非指导业务逻辑的方法中，进而改变这些行为的时候不影响业务逻辑的代码。

Aspects是AOP思想在iOS下的实现，写这篇文章主要是阅读Aspects源码，查看它具体实现过程。

Aspects库定义

在分析源码之前，首先看看Aspects文件中定义的一些信息：

AspectOptions

AspectOptions是一个枚举类型，用来说明hook的方法的调用时机。

AspectInfo

AspectInfo是对NSInvocation的一层包装，内部封装了3个属性，便于直接读取到NSinvocation对象、接收者和参数信息。

AspectIdentifier

AspectIdentifier代表了一个具体的Aspect内容，包括：被hook的原方法名SEL，hook的block和其签名、以及其调用时机。

AspectContainer

AspectContainer可以理解为aspect的容器类，单个的aspect对象都会被加入到该类所定义的集合中管理。

AspectTracker

AspectTracker用于跟踪记录被替换的类方法，防止重复替换。

行为总结

Aspects通过它提供的API可以在原方法执行前或执行后，来嵌入一个block来执行器内部封装的代码，而它内部使用了runtime来实现这个行为的：

（1）利用MethodSwizzle修改原方法的IMP，让原方法的实现走消息转发流程，也就是调用forwordingInvocation，并且将原方法的方法名添加一个新的前缀生成aliasSelector，让其IMP指向原方法的IMP，这样就完成了方法交换。

（2）因为forwordingInvocation在NSObject下默认的实现只是抛出错误，所以在原方法进行方法交换后，又利用MethodSwizzle修改了forwordingInvocation:方法的IMP，在其内部按照调用顺序执行传入的block代码段和原来的selector对应的方法。

（3）传入的block参数是不确定的，内部通过NSInvocation来调用这个不定参数的block：block底层结构也是结构体的形式，从结构体中取出编码的signature字段，从而创建NSMethodSignature（创建之后需要进行参数比较，确定block参数与原方法的参数保持一致），根据NSMethodSignature从而创建NSInvocation，其target为block，设置参数并调用；

####注意的地方

（1）对于实例方法的hook有限制条件，retain,release,autorelease,forwardInvocation是不能够被hook的。
（2）类方法因为被类对象共有，所以在类方法的继承链上只能被hook一次。继承下来的类方法被hook过不能再次被hook，子类hook过也不能hook，只有当前类可以修改hook。而这个实现是通过AspectTracker来完成的，如果其selectorNames集合中有存有改方法名，则证明被hook过，并将其父类的AspectTracker的集合set中添加子类tracker，以selectorName为键，证明子类hook了。
